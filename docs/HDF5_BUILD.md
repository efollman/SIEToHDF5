# HDF5 Build â€” Static Linking from Source

This document explains how HDF5 2.1.1 is compiled from C source and statically linked into the `sie2hdf5` binary using the Zig build system. The goal is **full portability** â€” the resulting binary has zero runtime dependency on an installed HDF5 library.

## Overview

Instead of relying on a system-installed `libhdf5`, this project:

1. Fetches the HDF5 2.1.1 source tarball as a Zig build dependency
2. Compiles ~291 C source files using Zig's built-in C compiler (clang-based)
3. Links the resulting static library into the final executable

This approach works identically for native builds and cross-compilation across all five supported targets.

## Dependency Declaration

In `build.zig.zon`, the HDF5 source is declared as a tarball dependency:

```zig
.hdf5 = .{
    .url = "https://github.com/HDFGroup/hdf5/archive/refs/tags/2.1.1.tar.gz",
    .hash = "N-V-__8AAEJlCw6diB9w3mfIldAS5c2K-8vKwhIkeCAagIgi",
},
```

Zig downloads and caches this tarball automatically on first build.

## Build Function

The `buildHdf5Lib()` function in `build.zig` constructs the static library:

```zig
fn buildHdf5Lib(b, hdf5_dep, target, optimize) *Compile {
    const hdf5_lib = b.addLibrary(.{
        .name = "hdf5",
        .linkage = .static,
        .root_module = b.createModule(.{
            .target = target,
            .optimize = optimize,
            .link_libc = true,
        }),
    });

    hdf5_lib.root_module.addCSourceFiles(.{
        .root = hdf5_dep.path("src"),
        .files = &hdf5_c_sources,     // ~291 files
        .flags = &.{ "-std=c11", "-w", "-D_GNU_SOURCE" },
    });

    // Build-settings stub (provides H5build_settings[] symbol)
    hdf5_lib.root_module.addCSourceFile(.{
        .file = b.path("hdf5_config/H5build_settings.c"),
        .flags = &.{ "-std=c11", "-w", "-D_GNU_SOURCE" },
    });

    // Include paths
    hdf5_lib.root_module.addIncludePath(hdf5_dep.path("src"));
    hdf5_lib.root_module.addIncludePath(hdf5_dep.path("src/H5FDsubfiling"));
    hdf5_lib.root_module.addIncludePath(b.path("hdf5_config"));

    return hdf5_lib;
}
```

### Key details

| Setting | Value | Reason |
|---|---|---|
| `linkage` | `.static` | Produces a `.a` archive â€” no shared library |
| `link_libc` | `true` | HDF5 C source requires libc (malloc, file I/O, etc.) |
| `-std=c11` | C11 standard | Required by HDF5 2.x |
| `-w` | Suppress warnings | HDF5 generates many warnings under clang; safe to suppress |
| `-D_GNU_SOURCE` | GNU extensions | Exposes POSIX functions like `pread`, `qsort_r`, `strtok_r` on Linux/glibc targets |

### Include Paths

Three include paths are configured:

1. **`hdf5_dep.path("src")`** â€” Main HDF5 headers (`H5public.h`, `H5private.h`, etc.)
2. **`hdf5_dep.path("src/H5FDsubfiling")`** â€” Subfiling VFD header (`H5FDsubfiling.h`), included by `hdf5.h` even when subfiling is disabled
3. **`b.path("hdf5_config")`** â€” Our custom configuration files (`H5pubconf.h`, `H5build_settings.c`)

## Configuration Header â€” `hdf5_config/H5pubconf.h`

The most critical piece of the static build is `H5pubconf.h`. This header is normally generated by CMake during an HDF5 build. We provide a hand-crafted version that configures HDF5 for all five target platforms.

### Platform Detection

The header uses standard compiler-defined macros to detect the target:

```c
#if defined(_WIN32)
  #define H5_HAVE_WINDOWS     1
  #define H5_HAVE_WIN32_API   1
  #define H5_HAVE_MINGW       1
  #define H5_HAVE_WINDOW_PATH 1
#endif

#if defined(__APPLE__)
  #define H5_HAVE_DARWIN 1
#endif
```

### Static Build Flag

```c
#define H5_BUILT_AS_STATIC_LIB 1
```

This disables `__declspec(dllimport/dllexport)` on Windows, which is essential for a static library build.

### Type Sizes

All targets are 64-bit, but type sizes differ across ABIs:

| Type | Linux LP64 | macOS LP64 | Windows LLP64 |
|---|---|---|---|
| `long` | 8 | 8 | **4** |
| `size_t` | 8 | 8 | 8 |
| `off_t` | 8 | 8 | **4** |
| `long double` | 16 (x86_64), 16 (aarch64) | 16 (x86_64), 8 (aarch64) | 8 |

The Windows LLP64 ABI is particularly important: `sizeof(long) = 4` and `sizeof(off_t) = 4` affect HDF5's internal assertions and type handling. HDF5's `H5win32defs.h` remaps `HDoff_t` to `int64_t` when `H5_HAVE_WIN32_API` is defined, handling the small `off_t`.

### Windows-Specific Handling

When `H5_HAVE_WIN32_API` is defined, HDF5's own `H5win32defs.h` header provides critical remappings:

| POSIX Function | Windows Replacement |
|---|---|
| `dlopen` / `dlsym` / `dlclose` | `LoadLibrary` / `GetProcAddress` / `FreeLibrary` |
| `qsort_r` | `HDqsort_context` (custom wrapper around `qsort_s`) |
| `strndup` | `H5_strndup` (internal reimplementation) |
| `ftruncate` | `_chsize_s` |
| `struct stat` / `fstat` | `struct _stati64` / `_fstati64` |

### POSIX vs Windows Function Availability

Functions only available on POSIX (guarded by `#if !defined(_WIN32)`):
- `alarm`, `asprintf`, `clock_gettime`, `fcntl`, `flock`, `fork`
- `fseeko`, `getrusage`, `gettimeofday`, `ioctl`, `pread`/`pwrite`
- `strcasestr`, `symlink`, `vasprintf`, `waitpid`

Functions available on both (including mingw):
- `strdup`, `tmpfile`, `gethostname`, `fseeko`

### Disabled Features

The following features are disabled in `H5pubconf.h` to minimize the build surface:

- **MPI / parallel HDF5** â€” No `H5_HAVE_PARALLEL`
- **Threading / threadsafe** â€” No `H5_HAVE_THREADS`, `H5_HAVE_THREADSAFE`
- **Compression** â€” No zlib (`H5_HAVE_FILTER_DEFLATE`), no szip (`H5_HAVE_FILTER_SZIP`)
- **Optional VFDs** â€” No ROS3, mirror, direct, subfiling, IOC, HDFS
- **MAP API** â€” No `H5_HAVE_MAP_API`
- **Language bindings** â€” No Fortran, C++, Java

Files for disabled features are still compiled but become empty translation units (all code gated behind `#ifdef` checks).

## Build Settings Stub â€” `hdf5_config/H5build_settings.c`

HDF5 references a global symbol `H5build_settings[]` that is normally generated by CMake. We provide a minimal stub:

```c
#include "H5private.h"

const char H5build_settings[] =
    "       SUMMARY OF THE HDF5 CONFIGURATION\n"
    "       =================================\n"
    "\n"
    "HDF5 Version: 2.1.1\n"
    "Built with Zig build system (SIEtoHDF5 project)\n"
    "Static library, no MPI, no threading, no compression filters\n"
;
```

## Linking

Every Zig module that uses HDF5 (either directly via C extern declarations or transitively) must have `link_libc = true` set. This is required because the HDF5 extern function declarations need the C ABI to resolve correctly:

```zig
// In build.zig
lib_mod.link_libc = true;
exe.root_module.linkLibrary(hdf5_native);
exe.root_module.link_libc = true;
```

## Source Files

The complete list of ~291 C source files is maintained in the `hdf5_c_sources` array in `build.zig`. These were derived from HDF5's `src/CMakeLists.txt` and include all modules:

- **Core**: H5, H5checksum, H5dbg, H5mpi, H5system, H5timer, H5trace
- **Attributes**: H5A*
- **Cache**: H5AC*, H5C*
- **B-trees**: H5B*, H5B2*
- **Datasets**: H5D*
- **Error handling**: H5E*
- **File management**: H5F*
- **File drivers**: H5FD* (core, sec2, stdio, log, multi, family, onion, splitter, etc.)
- **Groups**: H5G*
- **Heap**: H5HF*, H5HG*, H5HL*
- **IDs**: H5I*
- **Links**: H5L*
- **Memory**: H5MM*, H5FL*
- **Object headers**: H5O*
- **Property lists**: H5P*, H5PL*
- **References**: H5R*
- **Radix trees**: H5RT*
- **Dataspace**: H5S*
- **Shared messages**: H5SM*
- **Datatypes**: H5T*
- **Volume connectors**: H5VL*
- **Free space**: H5FS*
- **Event sets**: H5ES*
- **Maps**: H5M*
